# AGENT.md
## Рабочие договоренности проекта "Планка"

### 1. Контекст и миссия
- "Планка" — устойчивый цифровой инструмент коммуникации, а не обычное приложение.
- Цель: предсказуемая, надежная и автономная коммуникация для Андрюши.
- Базовый источник продуктовых решений: `CONTEXT.md`.

### 2. Важные личные поинты
- Андрюша — любимый сын заказчика проекта.
- Заказчик/разработчик: IT-инженер с 25-летним стажем.
- Основной стек заказчика:
  - Node.js
  - Нативный JavaScript / ES6
  - Веб-верстка
  - MongoDB
  - SQL (разные диалекты)
  - Go
  - Python

### 3. Инженерные принципы (обязательно)
- Надежность > фичи.
- Простота > гибкость.
- Контроль > магия.
- Предсказуемость > "умные" функции.
- Никаких скрытых жестов в MVP.
- Никакого автоисправления по умолчанию.
- Никаких неожиданных переходов интерфейса.
- Полная защита kiosk-режима.

### 4. Git-стратегия (когда становится критично)
- Критичный момент для создания git-репозитория: до начала регулярных кодовых изменений и до первой серии фич/фикс-коммитов.
- После инициализации: фиксировать все дельты атомарными коммитами.
- Базовые ветки:
  - `main` — стабильная ветка для поставки обновлений на устройство через родительский контроль.
  - `development` — рабочая интеграционная ветка для новых фич, багфиксов и экспериментов.
- Рекомендуемый поток:
  - `feature/*` и `bugfix/*` от `development`.
  - Слияние в `development` после проверки.
  - Релизное слияние `development` -> `main` только для проверенных изменений.

### 5. CI/CD (план)
- После появления тестового контура подключить CI/CD.
- Минимум для старта:
  - Прогон тестов и линтеров на PR в `development`.
  - Отдельный релизный pipeline для `main`.

### 6. Коммуникация в работе
- Общение: на "ты".
- Язык: русский.
- Тон: одобряюще-продуктивный, по делу, без лишней воды.

### 6.1. Язык в коммитах и PR
- Все коммиты и тексты PR — по-русски.
- Формат коммита: `PL-xxx: ...` или `ISSUE-<n>: ...` (императив, 72 симв. строка).
- Формат заголовка PR: `PL-xxx ...` или `ISSUE-<n> ...`.
- Тело PR включает блоки: краткое описание, состав изменений, критерии приёмки (чекбоксы), шаги QA, примечания.

### 6.2. Приватность локальных путей (обязательно)
- Никогда не публиковать в репозитории и GitHub-ресурсах (Issue, PR, комментарии, workflow summary) абсолютные локальные пути от корня системы.
- Запрещены к публикации любые пути вида `/Users/...`, `/private/var/...` и другие пути, раскрывающие локальные данные владельца машины.
- Для документации и переменных использовать только безопасные формы:
  - `~/...`
  - `<HOME>/...`
  - `/absolute/path/to/...` (обезличенный плейсхолдер)
- Перед коммитом и отправкой текстов в GitHub выполнять проверку на утечки путей и при срабатывании заменять значения на безопасный формат.

### 7. Обязательный git-flow работы
- После любой завершенной серии изменений в `development` создаем PR в `main`, а не ограничиваемся только коммитом.
- Я отправляю ссылку на PR для ревью и ручного merge в GitHub интерфейсе.
- После подтверждения merge ты пишешь `merged`, и я синхронизирую локальные ветки (`main`, затем `development`).
- После `merged` по умолчанию не запускаю отдельный ручной пост-контроль статусов задач и workflow.
- Закрытие карточек (`PL-xxx` или `ISSUE-<n>`) выполняется автоматикой GitHub Actions; вручную вмешиваюсь только если ты сообщил о проблеме.
- `TODO.md` и `CHANGELOG.md` синхронизирую в рамках следующей рабочей дельты или по явному запросу.
- Этот flow считаем оптимальным и используем по умолчанию для проекта.
- Разрешенные авто-действия без отдельного согласования:
  - коммиты и push в `development`;
  - создание и обновление PR `development` -> `main`;
  - `git fetch/pull` для синхронизации;
  - fast-forward merge `main` -> `development` после merge PR.
- Обязательное согласование перед действием:
  - любые удаления файлов/директорий;
  - сложные merge/rebase/cherry-pick с риском конфликтов;
  - любые потенциально деструктивные git-операции.
- Техническое примечание: если среда выполнения запросит системное подтверждение новой команды/префикса, это инфраструктурное ограничение; в проектной логике это считается заранее одобренным автоматическим git-flow.
- Для GitHub Project:
  - получать элементы и их статусы по project item и полям проекта, без хардкода конкретных item IDs;
  - редактирование project item (`gh project item-edit`) считать частью стандартного flow и выполнять автоматически.
- Для git/PR flow:
  - `git add`, `git commit`, `git push` в рамках текущих задач выполнять автоматически;
  - создание/просмотр/list PR (`gh pr create`, `gh pr view`, `gh pr list`) выполнять автоматически.
- Предпочтение по командам:
  - использовать `gh ... --jq` вместо пайпов с `jq`, где возможно, чтобы уменьшить инфраструктурные подтверждения.
- Список стабильных командных префиксов и шаблонов хранится в `COMMAND_TEMPLATES.md` и считается рабочим стандартом.
- Regex-маски в `COMMAND_TEMPLATES.md` считаются allowlist для автоматического flow-команд.
- По умолчанию использовать `scripts/codex/run.sh` как единую точку входа для git/gh операций; прямые команды использовать только как fallback.
- Для подготовки входных данных runner-а использовать только отдельные вызовы `scripts/codex/run.sh write/append/clear/copy`; не использовать chain-команды (`&&`, `;`, heredoc) для рабочего flow.
- При необходимости нового типа команды:
  - сначала добавляем шаблон в `COMMAND_TEMPLATES.md`;
  - затем используем в рабочем цикле.

### 8. Потоковая разработка (stream)
После сообщения `merged`:

1) Синхронизирую ветки (`scripts/codex/run.sh sync_branches`).
2) Для daemon-flow задача на запуск задается переводом `Status` из `Backlog` в `Todo`.
3) Демон мониторит `Status=Todo`, подхватывает задачу и переводит ее в `In Progress` (ACK).
4) После подхвата:
   - запускается автономный executor (`codex exec`) и выполняет реализацию в рамках Issue.
   - executor коммитит русскими сообщениями и обновляет PR по мере прогресса.
   - если есть блокер/вопрос, публикую комментарий в Issue через `scripts/codex/run.sh task_ask ...`;
   - если executor завершил прогон, но не финализировал задачу, публикую `AGENT_BLOCKER` в Issue и перехожу в ожидание твоего решения;
   - жду ответ в Issue-комментариях; daemon читает ответ и фиксирует `AGENT_RESUMED`.
5) По готовности:
   - выполняю `scripts/codex/run.sh task_finalize` (commit+push, create/update PR, `Status=Review`, `Flow=In Review`);
   - отправляю PR на ревью и прошу ревью;
   - двигаю статус задачи по flow.

Сетевой fallback:
- при проблемах DNS GitHub daemon дополнительно проверяет DNS/API Telegram;
- если Telegram доступен, daemon обязан отправить уведомление в бота о деградации и текущем режиме ожидания;
- если Telegram тоже недоступен, это явно отражается в `daemon_state_detail` и `daemon.log`.

Watchdog fallback:
- отдельный `watchdog`-процесс мониторит рассинхроны daemon/executor и зависшие lock-состояния;
- при деградации выполняет ступени восстановления (`soft` -> `medium` -> `hard`);
- все действия и причины фиксируются в `watchdog.log` и дублируются Telegram-сигналом (если канал доступен).

Flow Meta (зависимости):
- В задачах (`Issue`) поддерживается блок `## Flow Meta` с полями:
  - `Depends-On: #<issue>, #<issue>`
  - `Blocks: #<issue>, #<issue>`
  - `Auto-Queue-When-Unblocked: false|true`
  - `Execution-Mode: daemon|manual`
- Правило daemon-flow:
  - задача в `Todo` берется только если все `Depends-On` закрыты;
  - при незакрытых блокерах daemon не стартует задачу и отражает причину в state/log.
