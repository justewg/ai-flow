# AGENT.md
## Рабочие договоренности проекта "Планка"

### 1. Контекст и миссия
- "Планка" — устойчивый цифровой инструмент коммуникации, а не обычное приложение.
- Цель: предсказуемая, надежная и автономная коммуникация для Андрюши.
- Базовый источник продуктовых решений: `CONTEXT.md`.

### 2. Важные личные поинты
- Андрюша — любимый сын заказчика проекта.
- Заказчик/разработчик: IT-инженер с 25-летним стажем.
- Основной стек заказчика:
  - Node.js
  - Нативный JavaScript / ES6
  - Веб-верстка
  - MongoDB
  - SQL (разные диалекты)
  - Go
  - Python

### 3. Инженерные принципы (обязательно)
- Надежность > фичи.
- Простота > гибкость.
- Контроль > магия.
- Предсказуемость > "умные" функции.
- Никаких скрытых жестов в MVP.
- Никакого автоисправления по умолчанию.
- Никаких неожиданных переходов интерфейса.
- Полная защита kiosk-режима.

### 4. Git-стратегия (когда становится критично)
- Критичный момент для создания git-репозитория: до начала регулярных кодовых изменений и до первой серии фич/фикс-коммитов.
- После инициализации: фиксировать все дельты атомарными коммитами.
- Базовые ветки:
  - `main` — стабильная ветка для поставки обновлений на устройство через родительский контроль.
  - `development` — рабочая интеграционная ветка для новых фич, багфиксов и экспериментов.
- Рекомендуемый поток:
  - `feature/*` и `bugfix/*` от `development`.
  - Слияние в `development` после проверки.
  - Релизное слияние `development` -> `main` только для проверенных изменений.

### 5. CI/CD (план)
- После появления тестового контура подключить CI/CD.
- Минимум для старта:
  - Прогон тестов и линтеров на PR в `development`.
  - Отдельный релизный pipeline для `main`.

### 6. Коммуникация в работе
- Общение: на "ты".
- Язык: русский.
- Тон: одобряюще-продуктивный, по делу, без лишней воды.

### 7. Обязательный git-flow работы
- После любой завершенной серии изменений в `development` создаем PR в `main`, а не ограничиваемся только коммитом.
- Я отправляю ссылку на PR для ревью и ручного merge в GitHub интерфейсе.
- После подтверждения merge ты пишешь `merged`, и я синхронизирую локальные ветки (`main`, затем `development`).
- Этот flow считаем оптимальным и используем по умолчанию для проекта.
- Разрешенные авто-действия без отдельного согласования:
  - коммиты и push в `development`;
  - создание и обновление PR `development` -> `main`;
  - `git fetch/pull` для синхронизации;
  - fast-forward merge `main` -> `development` после merge PR.
- Обязательное согласование перед действием:
  - любые удаления файлов/директорий;
  - сложные merge/rebase/cherry-pick с риском конфликтов;
  - любые потенциально деструктивные git-операции.
- Техническое примечание: если среда выполнения запросит системное подтверждение новой команды/префикса, это инфраструктурное ограничение; в проектной логике это считается заранее одобренным автоматическим git-flow.
- Для GitHub Project:
  - получать элементы и их статусы по `Task ID` без хардкода конкретных item IDs;
  - редактирование project item (`gh project item-edit`) считать частью стандартного flow и выполнять автоматически.
- Для git/PR flow:
  - `git add`, `git commit`, `git push` в рамках текущих задач выполнять автоматически;
  - создание/просмотр/list PR (`gh pr create`, `gh pr view`, `gh pr list`) выполнять автоматически.
- Предпочтение по командам:
  - использовать `gh ... --jq` вместо пайпов с `jq`, где возможно, чтобы уменьшить инфраструктурные подтверждения.
- Список стабильных командных префиксов и шаблонов хранится в `COMMAND_TEMPLATES.md` и считается рабочим стандартом.
- Regex-маски в `COMMAND_TEMPLATES.md` считаются allowlist для автоматического flow-команд.
- При необходимости нового типа команды:
  - сначала добавляем шаблон в `COMMAND_TEMPLATES.md`;
  - затем используем в рабочем цикле.
