# PL-126 — Task Interpretation Layer / Intake Profile

Цель:
- ввести отдельный слой понимания задачи между свободной пользовательской постановкой и execution profile;
- отделить `task understanding` от `task execution`;
- сделать `micro` не режимом чтения сырой постановки, а режимом исполнения уже нормализованной задачи.

Проблема:
- текущий `micro-profile` уже умеет ограничивать execution path, telemetry и budget, но всё ещё слишком сильно зависит от качества сырой issue-постановки;
- свободная формулировка может содержать лишний narrative-text, mixed signals, `Вне scope`, implicit target files и ambiguous intent;
- classifier и context builder сейчас вынуждены извлекать execution-relevant смысл прямо из source text, что делает систему хрупкой.

Нужный инвариант:
- свободная постановка допустима на входе;
- но в `micro-executor` она не попадает как есть;
- execution layer работает только со `Standardized Task Spec`.

Новые уровни представления задачи:

1. `Source Definition`
- оригинальная формулировка пользователя / issue;
- хранится без переписывания;
- остаётся audit source и canonical human input.

2. `Standardized Task Spec`
- результат interpretation layer;
- содержит нормализованный execution contract:
  - `profileDecision`: `micro | standard | human_needed | blocked`
  - `interpretedIntent`
  - `candidateTargetFiles`
  - `expectedChange`
  - `checks`
  - `confidence`
  - `notes`
  - `rationale`

Task Interpretation Layer должен:
- прочитать свободную задачу;
- сделать repository-aware scoping;
- выбрать execution profile;
- нормализовать задачу;
- не выполнять саму задачу.

Границы слоя:
- Interpretation layer не редактирует код;
- не вызывает finalize path;
- не пишет commit/PR metadata;
- не запускает execution checks как часть business completion;
- не подменяет user intent, а только строит execution-ready spec.

Предлагаемые артефакты:
- `source_definition.json`
  - `taskId`
  - `issueNumber`
  - `title`
  - `body`
  - `replyText`
  - `capturedAt`
- `standardized_task_spec.json`
  - `taskId`
  - `issueNumber`
  - `profileDecision`
  - `interpretedIntent`
  - `candidateTargetFiles`
  - `expectedChange`
  - `checks`
  - `confidence`
  - `notes`
  - `rationale`
  - `sourceHash`
  - `repositoryContext`
- `intake_profile.json`
  - lightweight summary for runtime header / telemetry

Repository-aware scoping:
- interpretation layer может использовать deterministic repo facts:
  - file existence
  - file ownership / directory patterns
  - allow/deny maps for `micro`
  - known toolkit / infra / runtime / auth / finalize boundaries
- но не должна запускать сам executor logic и не должна эмулировать full implementation.

Профили interpretation outcome:
- `micro`
  - задача узкая, локальная, с высокой уверенностью и явным target scope;
- `standard`
  - задача исполнима, но scope шире или требует более свободного reasoning;
- `human_needed`
  - есть недостающий факт, выбор или ambiguous intent, который нельзя безопасно нормализовать;
- `blocked`
  - задача конфликтует с policy, rollout, budget, repo boundaries или требует explicit denial.

Связь с execution layer:
- `executor_start` сначала должен получать verdict interpretation layer;
- `micro_task_classifier` в текущем виде должен стать частью intake layer, а не единственным механизмом profile decision;
- `context_builder` должен строить prompt уже не из raw issue body, а из `standardized_task_spec.json`.

Нужный sequencing:

1. Intake artifacts
- materialize `source_definition.json`
- materialize `standardized_task_spec.json`
- materialize `intake_profile.json`

2. Intake decision
- interpreted intent
- profile decision
- confidence / rationale
- normalized target files / checks

3. Execution handoff
- `micro` path получает только normalized spec + deterministic file context;
- `standard` path получает normalized spec + broader execution context;
- `human_needed` path идёт в explicit ask layer;
- `blocked` path не стартует execution вообще.

Что должно уйти из execution layer после внедрения:
- parsing raw issue sections как основного источника intent;
- `micro`-specific interpretation через эвристики на сыром тексте;
- смешение classifier, scoping и execution budget logic в одном месте.

Definition of done для эпика:
- в v2 появляется отдельный intake layer с двумя canonical task representations;
- `micro-profile` больше не принимает raw issue body как execution contract;
- profile decision, target files, checks и expected change materialize-ятся до старта executor;
- `human_needed` и `blocked` становятся первичными outcomes interpretation, а не побочными отказами уже внутри execution path.
