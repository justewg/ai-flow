Что зафиксировано:
- выделен отдельный v2-эпик `Task Interpretation Layer / Intake Profile`;

  - `Source Definition`
  - `Standardized Task Spec`
- зафиксировано, что `micro-profile` — это execution mode нормализованной задачи, а не режим понимания свободной постановки;
- описаны intake outcomes:
  - `micro`
  - `standard`
  - `human_needed`
  - `blocked`

Новые артефакты эпика:
- `source_definition.json`
- `standardized_task_spec.json`
- `intake_profile.json`

Ключевой контракт:
- свободная постановка задачи допустима на входе и хранится без переписывания;
- interpretation layer делает repository-aware scoping и normalization;
- execution layer получает только normalized task spec и не занимается первичным пониманием user intent.

Почему эпик нужен:
- live evolution `micro-profile` показала, что execution guardrails, budget и context builder уже работают, но system всё ещё слишком зависит от формы сырой issue-постановки;
- interpretation concerns нужно вынести в отдельный слой, чтобы:
  - stabilise profile decision;
  - нормализовать target files / checks / expected change;
  - снизить coupling между free-form issue text и executor behaviour.

Следующий шаг:
- разложить `PL-126` на implementation phases:
  1. intake artifacts and source capture;
  2. standardized task spec builder;
  3. repository-aware profile decision;
  4. executor handoff migration from raw issue body to normalized spec.
