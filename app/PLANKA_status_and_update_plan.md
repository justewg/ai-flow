## 5. Предлагаемая схема обновлений по сети

Нужны два разных канала обновления.

### Канал A. Конфиг и лёгкие изменения
Используется для:
- изменения долей экрана;
- включения/выключения функций;
- параметров клавиатуры;
- мелких текстовых и визуальных правок;
- feature flags.

Схема:
1. Приложение хранит локальную текущую конфигурацию.
2. По запросу пользователя или по кнопке «Проверить обновления» делает HTTPS-запрос к серверу.
3. Получает `manifest.json` с:
   - версией конфига;
   - changelog;
   - ссылкой на полный config payload;
   - SHA-256 хешами.
4. Если версия новее:
   - скачивает новый config payload;
   - валидирует хеш;
   - применяет локально без переустановки APK.
5. При ошибке остаётся на предыдущей рабочей версии.

### Канал B. Полное обновление APK
Используется для:
- нативных изменений;
- изменений shell-части;
- усиления kiosk-логики;
- обновления Android-кода.

Схема:
1. Приложение по кнопке или вручную запрашивает `manifest.json`.
2. Если доступна новая APK:
   - показывает экран «доступно обновление»;
   - скачивает APK во внутреннее хранилище/кэш;
   - проверяет SHA-256;
   - запускает системную установку поверх текущего приложения.
3. После установки пользователь возвращается в приложение.

Канонический security/update contract для этого канала зафиксирован отдельно в `docs/pl-070-android-apk-update-flow.md`:
- доверенным считается только APK с корректным `sha256`, правильным `packageName/versionCode` и непрерывным signing path;
- silent install и downgrade через manifest в MVP не поддерживаются;
- фактом успеха считается только следующий старт shell с новым фактическим `versionCode`.

## 6. Что важно заложить в серверный manifest

Канонический MVP-contract для Android shell, publish infra и операторского runbook зафиксирован отдельно в `docs/pl-068-android-manifest-contract.md`.

В нём зафиксированы как обязательный минимум:

- manifest как полный snapshot двух каналов (`apk` + `config`);
- версии приложения и конфига;
- `minSupportedAppVersion`;
- `apk/config url` и `sha256`;
- user-facing `message`;
- manual update check;
- failure policy и rollback semantics для config/APK.

Ниже остаётся только короткий иллюстративный пример структуры.

Рекомендуемая структура:
```json
{
  "appVersion": 3,
  "appVersionName": "0.3.0",
  "configVersion": 7,
  "minSupportedAppVersion": 2,
  "apk": {
    "url": "https://example.com/releases/planka-0.3.0.apk",
    "sha256": "..."
  },
  "config": {
    "url": "https://example.com/releases/config-7.json",
    "sha256": "..."
  },
  "message": "Обновлена логика fullscreen и поведение клавиатуры"
}
```

## 7. Какой сетевой флоу выглядит самым практичным

### Для первой рабочей версии
- основной источник правды — локальное приложение;
- проверка обновлений — только вручную по кнопке;
- автоскачивание APK без автоприменения;
- конфиг можно применять сразу после проверки;
- APK — только через подтверждённую установку.

### Для следующей версии
- добавить фоновую проверку manifest по расписанию;
- не навязывать установку;
- показывать родителю индикатор «доступно обновление».

## 8. Что должен делать локальный агент/codex-cli

На локальной машине:
1. Собирать новый APK.
2. Считать SHA-256 APK и конфигов.
3. Публиковать APK и manifest в HTTPS-директорию.
4. Инкрементировать версии.
5. При необходимости публиковать только новый config payload без выпуска APK.
6. Сообщать, что новая сборка готова к проверке с планшета.

При этом publish path должен соблюдать два канонических документа:
- `docs/pl-068-android-manifest-contract.md` — manifest schema и общая семантика каналов;
- `docs/pl-070-android-apk-update-flow.md` — trust/signature/hash contract для APK.

Минимальный publish skeleton для этого теперь фиксируется в `docs/pl-071-android-publish-infra.md` и скриптах:
- `scripts/android_release/render_manifest.sh`
- `scripts/android_release/publish_release.sh`

## 9. Практическое решение для старта

Рекомендуется начинать не с полного OTA-механизма, а с такой лестницы:

1. Локальная установка APK по USB.
2. Затем HTTPS manifest + ручная проверка обновлений.
3. Затем отдельный канал конфигов.
4. И только после этого — полуавтоматическая доставка новых APK.

Так меньше риск сломать рабочее поведение тестового устройства.
