# Handoff: полная сводка контекста (на 2026-02-24)

> Исторический handoff сохранён как снимок контекста на 2026-02-24. Канонический toolkit-layout с марта 2026 года живёт в `.flow/`: `.flow/scripts/*`, `.flow/docs/*`, `.flow/config/*`, `.flow/state/*`. Пути `scripts/codex/*` и `.tmp/codex/*` ниже нужно читать как legacy-обозначения прежнего layout.

## 1) Что это за проект и как им управляют
- Проект: `planka`, основной рабочий процесс автоматизирован через локальный daemon + watchdog + GitHub Project + GitHub Actions + Telegram.
- Ветка разработки: `development`, PR обычно `development -> main`.
- Основная цель: потоковая разработка задач из Project-доски с минимальным ручным вмешательством.
- Язык процесса: русский (комментарии, коммиты, body PR, вопросы/ответы).
- Формат общения с владельцем проекта: на `ты`.

## 2) Коммуникационный контракт (важно для нового чата)
- Всегда писать по-русски.
- Обращение на `ты`.
- Если задача в фоне: кратко сообщать текущий статус и следующий шаг.
- Если блокер/вопрос: писать комментарий в Issue (и по возможности сигнал в Telegram), а не “молчать”.
- Если вопрос от пользователя уточняющий (без запроса доработки): отвечать как на вопрос, не запускать продолжение работы автоматически.
- Для явной команды продолжения после блокера рекомендовано:
  - `CODEX_MODE: REWORK`
  - и ниже текст, что делать дальше.

## 3) Целевой flow задач (актуальная модель)
Используем столбцы/статусы:
1. `Backlog`
2. `Todo` (триггер на взятие в работу)
3. `In Progress`
4. `Review`
5. `Done`

Правила:
- Daemon берет задачу только из `Todo`.
- После claim переводит задачу в `In Progress`.
- По завершению разработки и созданию/обновлению PR переводит задачу в `Review`.
- После merge PR задача переводится в `Done` (post-merge automation).
- Только `Issue` участвуют в автоподхвате; `Draft` игнорируются.
- `Untracked` файлы не должны блокировать flow.
- `Tracked` незакоммиченные изменения блокируют daemon (`WAIT_DIRTY_WORKTREE_TRACKED=1`).

## 4) Оркестрация vs исполнение
Исторически было:
- сначала автоматизировали оркестрацию (взять задачу/сигналы/PR-переходы),
- затем добавили executor, который реально выполняет разработку в фоне.

Итог:
- теперь поток включает и оркестрацию, и реальное выполнение.
- проверялись итеративные сценарии “коммит1 -> вопрос -> ответ -> коммит2 -> финализация”.

## 5) PR/сигналы/Telegram
- Принята модель: `1 задача = 1 PR`.
- Коммитов внутри одного PR может быть несколько.
- PR body может обновляться по ходу работы.
- Telegram-сигналы дифференцируются:
  - промежуточные (`TEMP_PROGRESS`, не требующие немедленного review),
  - финальные (`FINAL_REVIEW`, требуется проверка/принятие).
- Дополнительно усилено информирование:
  - комментарии в Issue перед финализацией PR,
  - логирование неудачных попыток отправки в GitHub,
  - outbox при временной недоступности GitHub API.

## 6) Сетевые проблемы и что уже сделано
Были частые деградации связи из sandbox/runtime к GitHub API (при том, что host-система могла работать нормально).

Сделанные меры:
- retry/backoff в GitHub-вызовах;
- outbox для отложенной отправки действий в GitHub;
- health-check и явные state-маркеры в heartbeat;
- watchdog с многоступенчатым recovery (`soft -> medium -> hard`);
- Telegram-уведомления о деградации и восстановлении;
- добавлены локальные DNS resolver-правила для `github.com` и `telegram.org` на host;
- добавлен отдельный probe-скрипт для мониторинга сетевой стабильности хоста.

Важно:
- при отсутствии GitHub и наличии Telegram daemon должен уведомлять через Telegram о деградации;
- если недоступны оба канала, состояние фиксируется в логах.

## 7) Что уточняли по GitHub Project
- В работу берутся только `Issue`, не `Draft`.
- Если задача создана в UI GitHub Project как Issue, она может иметь вид `planka #N` (это нормально).
- Для зависимостей добавлен `Flow Meta` в Issue:
  - `Depends-On`
  - `Blocks`
  - `Execution-Mode`
  - и т.д.
- Важное ограничение: зависимость на `Draft` как правило невалидна для автоматики; зависимость должна ссылаться на реальный Issue.

## 8) Пакет задач APP-* (GitHub App auth)
Принято решение идти через GitHub App (вместо второго user-аккаунта) для отдельной авторизации автоматизации.

Сделано по направлению:

- описаны шаги APP-01, APP-02 и последовательность зависимостей APP-xx;
- добавлен локальный auth-service под Node.js + PM2 обвязка;

Принцип:

## 10) Эволюция диалога с executor через Issue-комментарии
Сигналы:
- `AGENT_QUESTION`
- `AGENT_BLOCKER`
- `AGENT_RESUMED`
- `AGENT_RESUMED_REVIEW`
- `AGENT_ANSWER`
- `AGENT_IN_REVIEW`

Текущее ожидаемое поведение:
- В `Review`:
  - комментарий-вопрос -> `AGENT_ANSWER` и остаемся ждать;
  - комментарий-доработка -> `AGENT_RESUMED_REVIEW` и возврат в `In Progress`.
- После `AGENT_BLOCKER`:
  - вопрос пользователя (напр. “опиши блокер”) -> `AGENT_ANSWER`, без автопродолжения;
  - явная команда продолжить (`CODEX_MODE: REWORK` / “продолжай”) -> `AGENT_RESUMED` и запуск продолжения.

## 11) Согласованные ограничения безопасности/приватности

## 14) Шаблоны “финальных” текстов для итераций (чтобы стиль не потерялся)
Для ответа в чате после итерации:
- `Сделано: <кратко>`
- `Состояние: <текущий state/что ждём>`
- `Дальше: <следующий шаг>`

Для комментария в Issue при уточняющем вопросе пользователя:
- `CODEX_SIGNAL: AGENT_ANSWER`
- `CODEX_MODE: QUESTION`
- Короткий ответ по сути вопроса.
- Текущий контекст (статус задачи, ссылка/идентификатор блокера, последняя ремарка executor если есть).
- Явная инструкция для продолжения:
  - `CODEX_MODE: REWORK`

Для комментария в Issue при возобновлении:
- `CODEX_SIGNAL: AGENT_RESUMED` (или `AGENT_RESUMED_REVIEW`)
- Ссылка на исходный comment id пользователя.
