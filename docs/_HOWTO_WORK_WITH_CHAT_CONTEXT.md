# HOWTO: как удерживать рабочий контекст чата в лимите

## Цель
Сохранять управляемый контекст в долгих инженерных сессиях, чтобы не терять рабочую память и не упираться в переполнение окна.

## Пороговые значения
1. Держать запас контекста: `left >= 15-20%`.
2. При `used >= 70%` делать checkpoint (обновлять handoff-файл).
3. При `used >= 80%` переходить в режим кратких ответов и коротких логов.
4. При `used >= 85%` открывать новый чат по handoff, не дожидаясь деградации.

## Операционный ритм
1. Обновлять handoff каждые 20-30 минут или после крупного изменения.
2. Хранить "живой" handoff-файл в `docs/`:
   - `docs/CHAT_HANDOFF_YYYY-MM-DD.md`
   - или `docs/CHAT_HANDOFF_LIVE.md`
3. В handoff фиксировать только необходимое:
   - цель;
   - что сделано;
   - что осталось;
   - риски/блокеры;
   - следующие 1-3 шага.
4. Разделять длинную работу на этапные чаты: исследование -> внедрение -> ревью.

## Правила экономии контекста
1. Просить короткий формат: "максимум 5 пунктов".
2. Читать большие файлы частями (`100-150` строк) и собирать summary.
3. Не вставлять сырые длинные логи в чат: только ошибки, вывод и итог.
4. После каждого крупного шага сразу обновлять handoff.

## Аварийный протокол (когда контекст близок к пределу)
1. Сформировать финальный handoff в файл:
   - цель;
   - прогресс;
   - TODO;
   - риски;
   - команды для продолжения.
2. Открыть новый чат.
3. Первый запрос в новом чате:
   - "Прочитай handoff и продолжай с шага N".

## Важно
Суммаризация уменьшает давление на окно контекста текущего чата, но не "возвращает" уже израсходованные лимиты 5h/weekly.
