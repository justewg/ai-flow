# ISSUE-376 План backlog для flow-конфигуратора

## Контекст

Issue `#376` не про реализацию очередного низкоуровневого shell-скрипта, а про верхнеуровневый конфигуратор, который сможет развернуть текущий flow в произвольном проекте без ручной сборки из отдельных команд.

Целевой сценарий из issue:

1. подготовить `.flow` в новом или существующем репозитории;
2. подключить shared toolkit как submodule;
3. провести пользователя через вопросы с дефолтами и подсказками, где брать значения;
4. сгенерировать `flow.env`, state/log layout и служебные артефакты;
5. установить daemon/watchdog и довести проект до проверяемого preflight/smoke состояния.

## Что уже есть в toolkit

В текущем `.flow/shared` уже присутствуют почти все базовые строительные блоки:

- `create_migration_kit` и `apply_migration_kit` для переноса toolkit и repo overlay;
- `profile_init` для создания `flow.env`, state-dir, launchd labels и install/preflight/bootstrap шагов;
- `onboarding_audit` для проверки окружения, toolkit, secrets и обязательных настроек;
- `daemon_install` и `watchdog_install` для установки сервисов;
- onboarding/runbook-документация в `.flow/shared/docs/flow-onboarding-quickstart.md` и смежных файлах.

## Главный gap

Сейчас onboarding существует как набор отдельных команд и документов. Не хватает единой оркестрации, которая:

- сама определяет, что уже подготовлено в проекте, а чего не хватает;
- задаёт вопросы в правильном порядке и с безопасными дефолтами;
- объясняет происхождение каждого значения (`PROJECT_ID`, `PROJECT_NUMBER`, токены, labels, URLs);
- умеет материализовать `.flow` и shared submodule без ручной подготовки;
- запускает существующие низкоуровневые команды идемпотентно и собирает итоговый отчёт.

Из этого следует, что задача слишком широкая для одной implementation-дельты и должна быть разбита минимум на четыре связанные backlog-задачи.

## Предлагаемое разбиение

### 1. Контракт и сценарий визарда

Нужно сначала описать сам продукт визарда:

- режимы: fresh repo, partially configured repo, reconfigure;
- шаги мастера и точки возобновления;
- перечень вопросов, дефолтов и справок;
- формат промежуточного state-файла визарда;
- правила идемпотентности и safe-overwrite.

Без этого следующие задачи будут расходиться по требованиям и переизобретать входные данные `profile_init`/`onboarding_audit`.

### 2. Bootstrap `.flow` и shared toolkit

Отдельный слой нужен для materialize-подготовки проекта:

- создание каталога `.flow` и базовой структуры;
- подключение `/.flow/shared` как git-submodule;
- проверка/инициализация нужных symlink и директорий;
- подготовка безопасного шаблона `flow.env` и repo overlay.

Это самостоятельная техническая фаза, не зависящая от UX конкретного опросника.

### 3. Интерактивный опросник и генерация конфигурации

Следующий слой отвечает за человеко-ориентированный интерфейс:

- последовательность вопросов по GitHub repo/project, auth, Telegram, launchd/PM2 и optional сервисам;
- дефолтные значения и пояснения, где их брать;
- генерация/обновление `.flow/config/flow.env`;
- запись только подтверждённых пользователем значений;
- повторный запуск без разрушения уже заполненной конфигурации.

### 4. Install/preflight/smoke orchestration

Финальный слой должен доводить конфигурацию до рабочего состояния:

- запуск `onboarding_audit`;
- запуск `profile_init preflight/install`;
- установка daemon/watchdog;
- проверка health/status snapshot;
- финальный summary с перечислением того, что готово, что осталось вручную и какие smoke-команды запускать.

## Почему именно так

- это совпадает с уже существующим разделением в toolkit: bootstrap, config, install и audit уже живут в разных скриптах;
- каждая задача остаётся проверяемой и reviewable в одном PR;
- зависимости линейные только там, где они действительно нужны;
- future wizard можно реализовать как тонкую orchestration-надстройку над уже существующими командами, а не как новый монолит.
