# PL-002: Architecture Spike (Native Android vs WebView + Native Shell)

## Цель spike
- Зафиксировать архитектурное решение для MVP-прототипа с учетом ключевого риска: kiosk-надежность.

## Контекст решения
- Проекту критичны: предсказуемость, отсутствие случайных состояний, защита от выхода.
- По `CONTEXT.md` финальный выбор делается после прототипирования клавиатуры и kiosk-поведения.

## Варианты

### A. Native Android
Плюсы:
- Максимальный контроль над kiosk-ограничениями.
- Минимальные риски по системным жестам и фоновой устойчивости.

Минусы:
- Отдельный стек/темп разработки.
- Более высокая стоимость итераций UI на раннем этапе.

### B. WebView + Native Shell
Плюсы:
- Быстрее старт UX-прототипа клавиатуры.
- Проще и быстрее правки интерфейса.

Минусы:
- Kiosk-поведение зависит от качества нативного shell.
- Выше риск нестабильного поведения в edge-кейсах Android OEM.

## Критерии принятия решения (обязательные)
1. Невозможность случайного выхода из режима использования.
2. Восстановление состояния после перезапуска без потери последнего ввода.
3. Стабильная работа 50/50 экрана и клавиатуры без "скачков" UI.
4. Детект наклона не ломает ввод и не влияет на клавиатуру.
5. Поведение идентично после 50+ циклов sleep/wake.

## План проверки (короткий)
1. Собрать минимальный прототип клавиатуры и текстового поля в варианте B.
2. Провести сценарные прогоны kiosk-ограничений на целевом Android-устройстве.
3. Зафиксировать результаты в таблице pass/fail по критериям.
4. Если есть провал хотя бы по одному обязательному критерию kiosk-надежности, перейти на вариант A.

## Предварительное решение
- Для скорости MVP-исследования: стартуем с `WebView + Native Shell`.
- Зафиксированный fallback: при любом провале kiosk-критериев переходим на `Native Android`.
- Решение считается финальным только после прохождения критериев проверки.
